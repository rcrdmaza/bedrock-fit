*Prepared April 24, 2026 · Snapshot of `main` at commit `c90cffd`*

## 1. What's been built so far

### Stack

A deliberately small footprint — six runtime deps, no ORM-heavy abstractions, no auth library.

| Layer | Choice | Why it's interesting |
|---|---|---|
| Framework | Next.js 16.2.3 (App Router, Turbopack) | Server Components + Server Actions do most of the work; almost no client JS |
| UI | React 19.2.4 + Tailwind 4 | Stone palette, three-column header, zero component library |
| Database | Postgres via `postgres` 3.4.9 | Plain SQL driver, no connection-pool middleware |
| ORM | Drizzle 0.45.2 | Typed queries; no runtime schema validation |
| Hosting | Railway | Auto-deploy from `origin/main`; single Node service |
| Observability | Sentry (`@sentry/nextjs`) | Client + server error capture; source maps gated on env vars |
| Auth | Hand-rolled HMAC cookie | One admin password, 7-day signed session, constant-time compare |

Total source: **~2,300 LOC** across 35 TS/TSX files. Zero tests yet.

### Data model (`src/db/schema.ts`)

Four tables, all Postgres-native UUID PKs with `defaultRandom()`:

- **`athletes`** — identity row. `name`, unique `email`, optional `dob`/`gender`/`location`, an `xp` counter that's allocated but unused.
- **`results`** — one row per finisher per event. FK to athlete. Carries `eventName`, `eventDate`, `raceCategory`, `finishTime`, `overallRank`, `totalFinishers`, `percentile`, `bib`, `eventCountry`, plus claim tracking (`status`, `claimEmail`, `claimNote`, `claimSubmittedAt`). The event is implicit — no dedicated `events` table, the (name, date, category) triple is the key.
- **`eventMetadata`** — admin-curated layer on top. Same triple as a unique key. Optional `city`/`district`/`country`/`summary`/`routeUrl`/`routeImageUrl`. Lazily created when an admin first edits an event.
- **`eventPhotos`** — gallery rows, FK cascade to metadata, URL-only (no uploads). `sortOrder` for manual ordering.

### Feature map (what's actually shipped)

**Public, read-only**
- Leaderboards — filterable by distance category, paginated on a dedicated route, top-25 preview on the home page
- Recent-events photo carousel — vertical, one cover photo per event, centered above the leaderboards
- Events browser — searchable events list, per-event finisher table with ranked times, per-event "About / Route / Photos" tabs
- Athlete profile — race history, percentile, claim-your-result affordance
- Results browser — dual-mode (Results / Events) with name/bib/country/date filtering

**Athlete self-service**
- Claim single result (one form per row)
- Bulk-claim from an athlete profile (up to 20 rows, one submission)
- Both land in a `pending` queue for admin review — no email verification yet

**Admin (single password, `/admin`)**
- Pending claims review — grouped by submission, approve/reject per-row or per-batch
- Event management — edit metadata, manage photo gallery (add/remove/reorder)
- CSV import — 6-column required schema (`name`, `finish_time`, `overall_rank`, `gender`, `location`, `race_category`) plus optional `bib`, `event_country`. Chunked inserts so a 26k-row Boston CSV doesn't exceed Postgres's 65,535-param cap. Post-import flash redirects to `/results?view=events`.

**Operational**
- `/api/health` — liveness + DB probe with a 500ms timeout, suitable for Railway healthchecks
- Sentry tunnel route at `/monitoring` so ad-blockers don't drop client events
- `deploymentId` wired to `RAILWAY_DEPLOYMENT_ID` for version-skew protection — stale tabs now hard-reload instead of throwing "Failed to find Server Action"
- Server action body limit raised to 6 MB for real-race CSVs

### Architecture highlights worth calling out

- **No tests.** Every mutation runs through server actions; there's nothing testing the import parser, the claim flow, the bulk-approval query, or the HMAC cookie logic. Biggest structural risk in the codebase.
- **No event entity.** The (name, date, category) triple acts as a synthetic key everywhere. This keeps the importer simple — every row just carries the columns — but it means renaming an event requires a backfill, and you can't attach things like a registration link or a sponsor list without that join key reappearing on every `results` row.
- **Admin auth is a shared password.** Fine for a one-person product, structurally incompatible with the moment you want to (a) have more than one admin, (b) let race directors manage their own events, or (c) charge per user.
- **Single Railway instance.** No shared cache, no queue, no background worker. Fine at current scale; the import flow is synchronous and would block a request for minutes if someone tries a 100k-row CSV.

## 2. Site map

```
/                              Home. Leaderboard preview + recent-events carousel.
/?category=5K|10K|…            Chip-filtered home view.

/results                       Dual-tab browser (Results ↔ Events).
/results?view=events           Events tab preselected (used by import redirect).

/events?name=…&date=…&category=…
                               Single event detail. Tabs: Participants · Route · Photos.

/leaderboards/[category]       Full leaderboard for one distance, paginated.

/athletes/[id]                 Athlete profile + race history + bulk-claim.

/admin/login                   Password form, redirects to /admin on success.
/admin                         Pending-claims queue. Group by submission, approve/reject.
/admin/events                  Admin events list. Links into edit.
/admin/events/edit?name=…&date=…&category=…
                               Metadata form + photo manager.
/admin/import                  CSV upload + preview + commit.

/api/health                    JSON liveness probe for Railway/UptimeRobot.
/monitoring                    Sentry ingest tunnel (not user-facing).
```

No public auth routes; `Sign In` in the header is currently a placeholder.

## 3. Paths to profitability

Bedrock.fit today is a consumer-facing results site with single-admin CMS bolted on. Nothing in it currently generates revenue. Realistic paths, ordered by distance from what's shipped:

### Tier 1 — Ship in a quarter

**A. Race Director portal (SaaS).** The closest adjacency. Today one admin owns everything; split that into per-organization accounts and charge race directors a monthly fee ($29–$99/mo) to host their results, edit their own event page, and brand it with a logo/sponsor block. The existing admin surface is 80% of the product — what's missing is multi-tenancy on `eventMetadata` + a scoped auth model on top of the current HMAC cookie. You already have the CSV importer they'd use. Market comps: RunSignup results hosting, ChronoTrack's post-race page, Athlinks premium listings. **Best risk/reward ratio.**

**B. Sponsored event pages.** Much lighter lift. Add a `sponsor_name` / `sponsor_url` / `sponsor_logo_url` to `eventMetadata`, render a single unobtrusive sponsor stripe on the event detail page, sell those slots for $50–$500 per event to race-adjacent brands (nutrition, footwear, local PT clinics). Revenue ceiling is low but gross margin is ~100%. Useful as a wedge while you build the SaaS tier.

**C. Athlete premium.** Paid profile tier ($5–$10/mo) for heads-to-head comparison vs. friends, year-over-year progression charts, a shareable "season recap" export. Requires public auth (`/auth/sign-up`) — which is currently just a `href="#"` — but the data already exists in `results`. The stumbling block is that runners historically don't pay for recap features; Strava's free tier has eaten the obvious ground. Would need a hook other consumer running apps don't have.

### Tier 2 — Ship in 2–3 quarters

**D. Data API for coaches & shoe brands.** A paid read-only API over anonymized/aggregated finish-time data by event, distance, age group, country. Coaches pay $50–$200/mo for a benchmark firehose; shoe brands pay in the thousands. Needs: (1) enough volume to be useful (hundreds of events, tens of thousands of results), (2) legal scaffolding — terms and a PII stance — (3) a rate-limited API with API keys. Your schema is already close.

**E. Affiliate + referrals.** Race-entry affiliate links on event pages (RunSignup, Race Roster both have programs), gear affiliate on athlete profiles (Amazon, running-specialty retailers). Low single-digit percentage cuts, but on a content-heavy site with SEO traffic, it compounds. Needs real organic traffic first — which means content (see G below).

**F. Event registration layer.** You own results; the logical expansion is to also own registrations, charging 5–7% per sign-up (that's the market rate). Much bigger scope — payments, waivers, timing-chip integration — but it's where the money actually is in this space. Only attempt if A has traction.

### Tier 3 — Speculative

**G. Content / SEO moat.** Spin up a blog (the nav link is already there) with race recaps, course previews, training guides. Goal isn't direct revenue, it's cheap long-tail traffic that funnels into A/B/E. Cost is time, not engineering; LLM-assisted drafts with human polish make this tractable solo. Watch out for Google's stance on AI-generated content.

**H. White-label to timing companies.** A timing company currently hands clients a generic results page. Offer a hosted Bedrock.fit instance as their branded frontend for a flat license fee per event. Enterprise sales motion, not a self-serve thing.

### Things not worth doing (my read)

- **Display ads.** Running audiences don't click, CPMs are low, and ads cheapen a clean site. Affiliate beats it every time for this vertical.
- **Strava clone.** Don't try to displace Strava on segment leaderboards or training logs. Stay in the finish-line niche.
- **Mobile app.** Your use case is read-heavy and mostly post-race — the web is fine. Build a PWA shell before a native app.

## 4. Suggested sequencing

If I had to pick the next 6 weeks:

1. **Week 1–2:** add tests around the import + claim flows (biggest latent risk), fix the `react-hooks/set-state-in-effect` violation in `race-history.tsx`.
2. **Week 2–3:** implement public auth (NextAuth / custom) so the "Sign In" stub becomes real. This unlocks both the athlete premium path and multi-admin.
3. **Week 3–5:** scope the Race Director portal as a thin v1 — an organization has many events, each event has one or more admins, each admin sees only their own claims queue and import page. Use the existing schema plus an `organizations` table and an `event_owner_id` column on `eventMetadata`.
4. **Week 5–6:** ship sponsor stripes (feature B) and land one or two paying events to prove the billing loop works before investing in the SaaS portal.
